# Incident Priority Policy

## Purpose

This policy defines how IT incidents should be prioritized.

## Priority Levels

### P1 - Critical

Use P1 when a business-critical system is unavailable for many users, payroll or finance close is blocked for a team, a confirmed security incident affects multiple users, or no workaround exists.

Target response: Immediate.

### P2 - High

Use P2 when a single user is blocked from business-critical work, payroll or finance close is time-sensitive, a sensitive application cannot be accessed by an authorized user, or a high-risk phishing incident is reported by one user.

Target response: Same business day.

### P3 - Medium

Use P3 when the user can still work using a workaround, performance is degraded but not blocking critical work, a non-critical application has issues, or a software installation request is needed for normal work.

Target response: 1-2 business days.

### P4 - Low

Use P4 when the request is informational, the user asks a general how-to question, no active business impact exists, or the issue is cosmetic.

Target response: Best effort.

## Agent Instruction

The agent should use this policy to classify priority. The agent should explain the reason for the priority in tickets and audit logs.
